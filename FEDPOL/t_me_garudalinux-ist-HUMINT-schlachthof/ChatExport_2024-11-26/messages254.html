the partition format is not your issue. There are errors reported by the drive at SATA level. That means there are errors at the hardware level somewhere.

If the connection is fine and you say it must be a software problem, you should look into the direction of the sata controller drivers / firmware.
For example, try to boot with the LTS kernel or dig into kernel parameters or stuff...

Unless the SATA errors are not fixed, it will always be slow and you might expirience data loss. So that should be your priority.

Siddharth Kumar Jha
In reply to this message
The if [ -d ~/.local/bin ] first checks if the directory mentioned exists or not if it does it adds it to your path. A path is an environment variable that tells your terminal where all your commands are that your are trying to run. Eg: garuda-inxi or garuda-update are shell scripts located in /bin/ how would the terminal know where these commands are?? The answer is paths it searches those paths and executes them if they are not already present in the current working directory.

Next, [ -x /usr/bin/starship ] tests if starship is an executable or not if it is it loads starship your cross terminal prompt that helps your terminal look the way it looks.

Next, source /usr/share/.../ftc.bash
Another bash file this source command simply loads it up as we start the terminal after execution of the above commands

Then comes the aliases. They are shorthands for various commands eg:
alias ls='exa -al....'
Tells bash to run the exa command instead of the regular ls present in /bin/ls

And lastly the [ -x /usr/bin/fastfetch ] tests if fastfetch is executable or not if it is then it runs the command
fastfetch --load-config neofetch
Or whatever flag might be there for you.

At the end bashrc is just a regular script that runs whenever the bash terminal is launched.

Look there can be two scenarios with a dual boot system.
1. A windows update decided to remove grub from your system. Which means you can't access your linux installation at all and would need a live iso (aka the os you flashed in your usb stick) to repair your system by accessing it via chroot.

You can access tty without having to brick your system. On any linux installation the key combination of
Ctrl+alt+f2
Will land you into tty2 the gui session on your xorg server is on tty1 which you can access again via
ctrl+alt+f1.
