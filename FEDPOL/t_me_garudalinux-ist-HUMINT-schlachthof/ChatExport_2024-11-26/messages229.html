fast startup and fast boot are very different things in different contexts. fast boot (on windows 8) or fast startup (windows 10+) is something that basically puts your computer to sleep/hibernation (depending on power state settings and support) instead of fully shutting down, unless you're doing an update. this is a very bad thing when dual booting and you ALWAYS want it turned off for that reason. fast boot (occasionally called fast startup) in a BIOS/UEFI context refers to skipping some self-tests during the POST process to make it get to the stage of loading an OS faster.

to your last question:
You can just reclaim the space. Is it only another partition or another drive? See KDE Paritionmanager (if running kde, else something else like gpart). It should be easy to delete the partition there. You can then either create something new, linux readable or just add the space (or drive) to your existing btrfs filesystem (depens a bit on if it is just a partition or an extra drive on how to do that).

Regarding the time:
Usually linux keeps the BIOS time in UTC and calculates the local time from that itself. If you dual boot with windows, that will fail, because Windows sets BIOS time to local time.
if you go to garuda-settings, time & date, there is a setting (second checkbox, the one with realtime clock in the name) where you can define if linux should set the BIOS to local time (and what to assume on what it is set)

@Catra (Catraninu Felism√©) posted in A solution to an interesting problem. Multi-monitir, multi card: console framebuffer

A recent upgrade to the kernel can change which framebuffer the linux console uses if you have a multi-gpu/multi-monitor set up. 
If you need the console to appear on a different framebuffer (not, not which monitor, but which gpu) you add the following to GRUB_CMDLINE_LINUX_DEFAULT= 
fbcon=map:x 
Where x is the framebuffer you want to use. (Starts at 0) 
HTH, Additions and corrections would be cool.
